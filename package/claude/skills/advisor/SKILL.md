# advisor 相談プロトコル

主担当（あなた）が判断の難しい局面で、高位モデルの advisor subagent に助言を求めるためのプロトコル。
advisor は助言者であり、採否の決定と結果責任は主担当に残る。

## 起動形態

- 引数あり（`/advisor <質問>`）: その質問について今すぐ 1 回相談する
- 引数なし（`/advisor`）: 運用モードを宣言する。そのセッション中、下記トリガーに該当したら自律的に advisor へ相談する（セッションをまたいで持続はしない）

## 相談トリガー（運用モード時）

1. **設計・方針の分岐点**: アーキテクチャ選定やトレードオフ判断など、後戻りコストの大きい意思決定の前
2. **行き詰まり**: 同じ問題への修正が 2〜3 回失敗したとき。自力継続ではなく状況を整理して相談する
3. **成果物の最終チェック**: 実装完了・プラン確定などの節目でレビューを依頼する
4. **不確実性の自己申告**: 確信が持てないとき。ただし先に下記のルーティング規則を適用する

## 不確実性のルーティング規則

判定テスト: 「答えに必要な情報が、リポジトリ・ドキュメント・公開情報から入手可能か」。
入手可能なら advisor、人の頭の中にしかないなら AskUserQuestion で人間に聞く。

**人間に聞く — 情報が人の頭の中にしかないケース:**

- 要件の優先順位・スコープ判断（どこまで今回やるか、どの UX を望むか）
- ビジネス・組織の事情（リリース時期、他チームとの合意、運用体制）
- ユーザーの好み・過去の口頭決定など、記録に残っていない前提
- 破壊的操作や外部公開など、承認そのものが必要な行為

**advisor に聞く — 情報は入手可能だが評価が難しいケース:**

- アーキテクチャ・設計のトレードオフ評価（各案の材料は揃っている）
- 行き詰まったデバッグの原因仮説の妥当性判断
- 実装・プランの品質レビュー
- 技術選定の裏付け（オンライン調査を含む）

両方が混ざっている場合は先に人間へ聞いて前提を確定させ、そのうえで advisor に相談する。

## 相談の手順

1. 相談パッケージを組み立てる。advisor は会話コンテキストを持たないため、以下 5 点を必ず含める
   1. **問い**: 1 文で答えられる形に絞る
   2. **背景**: 何をしていて、なぜこの判断が必要か
   3. **制約**: 技術的制約、プロジェクト規約、既に確定した決定
   4. **検討済みの案と棄却理由**: 行き詰まり相談では試行と失敗内容
   5. **関連ファイルパス**: advisor が裏取りに使う。行番号付き推奨
2. Agent tool で呼び出す
   - `subagent_type: advisor`
   - `run_in_background: false`（回答を待って判断する場面のため同期実行）
   - `name` を付ける（例: `advisor-1`）。追加質問は SendMessage で同一 advisor に送り、コンテキストを維持する
   - モデルを変えたい場合のみ `model` パラメータで override する（デフォルトは agent 定義の fable）

## 回答の扱い

- 採否の決定と結果責任は主担当が持つ。advisor の回答を無検討で採用しない
- advisor と主担当の見解が割れた場合は、独断で選ばず両論と自分の推しをユーザーに開示する
- 相談したこと・回答の要旨は、ユーザー向けの報告に含める（相談が見えないと監査できないため）
